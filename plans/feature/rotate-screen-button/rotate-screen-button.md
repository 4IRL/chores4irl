## Summary

Add a user-facing "rotate screen" button to the chores4irl kiosk so display orientation can be changed without editing host config files over SSH. Because rotation is a **host** concern (kanshi / labwc / `wlr-randr` run in the Pi's Wayland session, outside Docker), the feature uses a host-bridge: the React button calls a new Express endpoint, the backend writes a `rotation.json` state file into a **dedicated rmilarachi-owned bind-mounted host directory** (separate from the SQLite `chores-data` named volume), and a new host-side systemd **user** service watches that file and applies the rotation live (`wlr-randr`) + persistently (kanshi config + labwc `calibrationMatrix`), keeping display and touch in sync. The bind mount is required because a non-root systemd **user** service cannot traverse the root-`0700` `/var/lib/docker/volumes/` tree where the named volume lives (DD-7). The button toggles between the two **portrait** orientations only (`90 ‚Üî 270`); landscape transforms (`0`/`180`) are intentionally unreachable from the button as a **deliberate portrait-first default** for the wall-mounted kiosk (DD-8 ‚Äî note the overlay-lockout justification was corrected; see DD-8). The rotate control is **housed in a new collapsible settings panel** (DD-5): a NavBar toggle that expands into an overlaid single-row banner of kiosk controls. Rotate is the one functional control; brightness, screen-blank, restart, undo, and redo ship as **disconnected, optimistic icon placeholders** for future work (logged in `plans/ledger/260630_future_feature_list.md`).

## Research Findings

- **Backend** (`backend/src/app.ts`): Express 5, ESM (`.js` import suffixes), no router abstraction ‚Äî routes wired inline in `app.ts`, data-access functions in per-concern modules (`chores.ts`). Every response uses `ApiResponse<T>` (`{ success, data?, error? }` from `types/SharedTypes.d.ts`) with the `satisfies` keyword; validation is manual `if`-guards (no zod); each route has its own `try/catch` (no global error middleware). DB path resolves via `process.env.DB_PATH ?? <fallback>` in `db.ts`; the data dir is `path.dirname(DB_PATH)` ‚Üí `/data`. **No `fs` import exists in `backend/src/` yet.**
- **Frontend** (`frontend/src/App.tsx`, `services/choreApi.ts`): All API calls use relative `/api/...` URLs through a shared `handleResponse<T>` wrapper that throws on `success:false`. `App.tsx` owns a single `error` state shown as a red banner (no toast lib). `lucide-react` is installed (`RotateCw`/`RefreshCw` available). Buttons use `type='button'`, `min-h-[44px]` touch targets, Tailwind dark theme, `aria-label` for icon-only. `NavBar` has no right-side slot today.
- **Request chain**: nginx proxies `location /api/` ‚Üí `http://backend:3000` with **no path rewriting** (route must be registered as `/api/display/...`). **No auth/CSRF/session** anywhere; CORS is wildcard. The backend is not published to the host ‚Äî only reachable via nginx ‚Äî but the endpoint is reachable by any LAN device on port 80 (small abuse surface; see Step 6).
- **Host bridge**: volume real path is `/var/lib/docker/volumes/chores4irl_chores-data/_data/` (project name inferred from the `chores4irl` directory ‚Äî **verify on the Pi**). Backend container runs as non-root `node` and can write `/data`. Host artifacts to update: `~/.config/kanshi/config` (persist), `~/.config/labwc/rc.xml` `calibrationMatrix` (persist), `wlr-randr --output <CONNECTOR> --transform <T>` (live) + `labwc --reconfigure`. **Critical naming distinction (per the corrected `deploy/pi/README.md`): `wlr-randr --output` takes the CONNECTOR NAME (e.g. `HDMI-A-1`/`HDMI-A-2`), NOT the `DZX Z3 ...` make/model/serial description ‚Äî only kanshi matches by description.** The connector name drifts between HDMI ports (the original deploy bug), so the agent must **discover it at apply time** from `wlr-randr`, never hardcode it. **kanshi edits are NOT live** ‚Äî only `wlr-randr` rotates immediately. The transform‚Üîmatrix table and current shipped values (`transform 90` ‚Üî `0 -1 1 1 0 0`) live in `deploy/pi/README.md`. **Critical constraint**: `wlr-randr`/`labwc --reconfigure` need the graphical session's `WAYLAND_DISPLAY` + `XDG_RUNTIME_DIR` ‚Äî a system service won't have them, so the watcher must be a **systemd user service** in the labwc session.
- **Tests**: vitest both ends. Backend uses supertest against the imported `app` with in-memory SQLite (`TEST_DB_PATH=':memory:'`); `db-path.test.ts` is the precedent for filesystem tests (`mkdtempSync` + `vi.resetModules()` + dynamic `import()` + env override + `rmSync`). Frontend uses Testing Library + `vi.stubGlobal('fetch', mockFetch)`. Playwright smoke specs use `page.waitForResponse()` / `page.route()`. The **host agent (bash) and physical rotation are not unit-testable** ‚Äî manual verification only. Commands: `npm run test --workspace backend`, `npm run test --workspace frontend`, `npx playwright test`.

## Design Decisions

- **Button behavior (DD-8)**: one press **toggles between the two portrait orientations `90 ‚Üî 270`** (a 180¬∞ portrait flip ‚Äî e.g. for an inverted wall mount). The button never produces the **landscape** transforms `0`/`180` ‚Äî a deliberate portrait-first default for the wall-mounted kiosk (see "Button orientation set" below; an earlier overlay-lockout rationale was corrected there). The backend endpoint stays a simple **absolute setter** (`POST /api/display/rotation` with `{ rotation: 0|90|180|270 }` ‚Äî the full set is still accepted for manual override / future use); the frontend reads current state on mount via `GET` and computes the next value on click as `next = rotation === 90 ? 270 : 90` (any non-`90` value, including a hand-set `0`/`180`, resolves to a portrait orientation on the next press). This keeps the endpoint trivially testable and the toggle logic in one place. (Alternative considered: a full 4-way cycle ‚Äî rejected because `0`/`180` land in landscape and trigger the overlay on a small kiosk panel.)
- **State ownership**: `rotation.json` in the rmilarachi-owned bind-mounted host directory is the single source of truth for "desired rotation," written by the backend and read by both the host agent (to apply) and the frontend `GET` (to seed the button). The host agent also rewrites kanshi config so the choice survives reboot.
- **Access model (DD-7)**: the agent must run as a systemd **user** service (it needs the labwc session's `WAYLAND_DISPLAY`/`XDG_RUNTIME_DIR`), but a non-root user cannot traverse the root-`0700` `/var/lib/docker/volumes/` tree where the named volume's `_data` lives. So `rotation.json` lives in a **dedicated bind-mounted host directory owned by the `rmilarachi` user** (`./chores-rotation` next to `docker-compose.yml`, mounted at `/rotation`), kept separate from the SQLite `chores-data` named volume. The container `node` user is uid 1000 ‚Äî the same uid as the Pi `rmilarachi` user (uid 1000 on this Pi ‚Äî **verify with `id -u rmilarachi`**, see Step 5 preflight) ‚Äî so the backend can atomically write the file and the user service can read it with no `sudo`, no ACL hacks, and no second forwarding service. This also decouples rotation state from the DB volume, so the DD-6 name-pin step no longer risks the rotation file (and the chores DB volume model is left unchanged ‚Äî no DB migration). (Rejected: Option 2 root system-service copy-hop ‚Äî extra service + poll loop; Option 3 `setfacl` on the volume ‚Äî fragile, Docker may reset ACLs.)
- **Button orientation set (DD-8)**: the app is portrait-first by explicit design ‚Äî `frontend/src/index.css` ships `#rotate-overlay` (full-screen `z-9999`) shown via `@media (orientation: landscape) and (max-height: 500px)`, a tested guard (`e2e/smoke.spec.ts`). **Correction to an earlier rationale:** the shipped panel is `1024√ó600`, so in *landscape* the viewport is `1024√ó600` ‚Üí height `600px`, which is **NOT** `‚â§ 500px` ‚Äî so the overlay does **not** actually fire in landscape on this panel, and transforms `0`/`180` would render a normal-but-sideways UI, **not** a lockout. The earlier "landscape blacks out the screen / locks the user out" justification was therefore wrong for this hardware. The button still **toggles only between the two portrait transforms `90 ‚Üî 270`**, but as a **deliberate conservative default**, not because `0`/`180` are unsafe: the kiosk is wall-mounted in portrait and the only orientation change it realistically needs is a 180¬∞ portrait flip (e.g. an inverted mount). The wire contract / type `DisplayRotation = 0|90|180|270` and `VALID_ROTATIONS` are kept intact (backend + agent still accept all four for manual override or a future landscape-capable layout), but the button's click handler never produces `0`/`180`. **Revisit if a different panel is used whose landscape height is `‚â§ 500px`** ‚Äî there the overlay *would* fire and portrait-only becomes a hard requirement rather than a default. (Rejected: Option 3 re-scope to a bespoke two-state toggle ‚Äî needlessly tears down the wire contract.)
- **Control surface (DD-5 ‚Äî REVISED to a settings panel)**: the rotate button is **not** placed bare in the NavBar. The NavBar `rightSlot` (the same real estate) instead holds a new **`SettingsPanel`** ‚Äî a single toggle button showing a `Settings` (gear) icon when collapsed and an `X` icon when open (`aria-expanded` reflects state) ‚Äî that expands into an **overlaid, single-row banner of icon buttons**, absolutely positioned/anchored to the toggle so it does **not** reflow the NavBar or chore list. The panel is the home for kiosk/device controls. **Only the rotate control is functional in this plan**; the rest are **disconnected, optimistic placeholders** ‚Äî rendered as real, `aria-label`led icon buttons with no-op handlers and a `data-placeholder='true'` marker (no backend wiring) so the surface looks complete and future work just connects them. Initial control set, left‚Üíright: **Rotate** (functional, `RotateCw`), **Brightness** (`Sun`), **Screen blank** (`MonitorOff`), **Restart** (`Power`), **Undo** (`Undo2`), **Redo** (`Redo2`). Calibration was explicitly excluded. The `rightSlot` mechanism itself is unchanged (DD-5, Option 2) ‚Äî only its occupant changes from `RotateScreenButton` to `SettingsPanel`, which then houses the rotate button. Placeholder future functionality is logged in `plans/ledger/260630_future_feature_list.md`. (Supersedes the earlier "single rotate button in `rightSlot`" decision.)
- **Transform vs matrix**: the wire contract carries only the integer transform (`0/90/180/270`). The host agent owns the transform‚Üítoken‚Üí`calibrationMatrix` lookup (from `deploy/pi/README.md`) via a single `case` table (DD-1). **Integer `0` maps to the keyword `normal`** for both `wlr-randr --transform` and the kanshi `transform` token ‚Äî both tools reject literal `0`. Frontend/backend never touch the token or the matrix; the integer stays on the wire only.
- **Security**: the endpoint inherits the app's current no-auth posture. The host agent **must allowlist-validate** the transform before interpolating it into `wlr-randr`/`sed` (injection guard). An optional shared-secret header is Step 6 (off by default).

# In-App Rotate-Screen Button (Pi Kiosk)

## Summary

Add a user-facing "rotate screen" button to the chores4irl kiosk so display orientation can be changed without editing host config files over SSH. Because rotation is a **host** concern (kanshi / labwc / `wlr-randr` run in the Pi's Wayland session, outside Docker), the feature uses a host-bridge: the React button calls a new Express endpoint, the backend writes a `rotation.json` state file into a **dedicated pi-owned bind-mounted host directory** (separate from the SQLite `chores-data` named volume), and a new host-side systemd **user** service watches that file and applies the rotation live (`wlr-randr`) + persistently (kanshi config + labwc `calibrationMatrix`), keeping display and touch in sync. The bind mount is required because a non-root systemd **user** service cannot traverse the root-`0700` `/var/lib/docker/volumes/` tree where the named volume lives (DD-7). The button toggles between the two **portrait** orientations only (`90 ↔ 270`); landscape transforms (`0`/`180`) are intentionally unreachable from the button because they trip the app's portrait-only `#rotate-overlay` and would black out the UI (DD-8).

## Research Findings

- **Backend** (`backend/src/app.ts`): Express 5, ESM (`.js` import suffixes), no router abstraction — routes wired inline in `app.ts`, data-access functions in per-concern modules (`chores.ts`). Every response uses `ApiResponse<T>` (`{ success, data?, error? }` from `types/SharedTypes.d.ts`) with the `satisfies` keyword; validation is manual `if`-guards (no zod); each route has its own `try/catch` (no global error middleware). DB path resolves via `process.env.DB_PATH ?? <fallback>` in `db.ts`; the data dir is `path.dirname(DB_PATH)` → `/data`. **No `fs` import exists in `backend/src/` yet.**
- **Frontend** (`frontend/src/App.tsx`, `services/choreApi.ts`): All API calls use relative `/api/...` URLs through a shared `handleResponse<T>` wrapper that throws on `success:false`. `App.tsx` owns a single `error` state shown as a red banner (no toast lib). `lucide-react` is installed (`RotateCw`/`RefreshCw` available). Buttons use `type='button'`, `min-h-[44px]` touch targets, Tailwind dark theme, `aria-label` for icon-only. `NavBar` has no right-side slot today.
- **Request chain**: nginx proxies `location /api/` → `http://backend:3000` with **no path rewriting** (route must be registered as `/api/display/...`). **No auth/CSRF/session** anywhere; CORS is wildcard. The backend is not published to the host — only reachable via nginx — but the endpoint is reachable by any LAN device on port 80 (small abuse surface; see Step 6).
- **Host bridge**: volume real path is `/var/lib/docker/volumes/chores4irl_chores-data/_data/` (project name inferred from the `chores4irl` directory — **verify on the Pi**). Backend container runs as non-root `node` and can write `/data`. Host artifacts to update: `~/.config/kanshi/config` (persist), `~/.config/labwc/rc.xml` `calibrationMatrix` (persist), `wlr-randr --output <CONNECTOR> --transform <T>` (live) + `labwc --reconfigure`. **Critical naming distinction (per the corrected `deploy/pi/README.md`): `wlr-randr --output` takes the CONNECTOR NAME (e.g. `HDMI-A-1`/`HDMI-A-2`), NOT the `DZX Z3 ...` make/model/serial description — only kanshi matches by description.** The connector name drifts between HDMI ports (the original deploy bug), so the agent must **discover it at apply time** from `wlr-randr`, never hardcode it. **kanshi edits are NOT live** — only `wlr-randr` rotates immediately. The transform↔matrix table and current shipped values (`transform 90` ↔ `0 -1 1 1 0 0`) live in `deploy/pi/README.md`. **Critical constraint**: `wlr-randr`/`labwc --reconfigure` need the graphical session's `WAYLAND_DISPLAY` + `XDG_RUNTIME_DIR` — a system service won't have them, so the watcher must be a **systemd user service** in the labwc session.
- **Tests**: vitest both ends. Backend uses supertest against the imported `app` with in-memory SQLite (`TEST_DB_PATH=':memory:'`); `db-path.test.ts` is the precedent for filesystem tests (`mkdtempSync` + `vi.resetModules()` + dynamic `import()` + env override + `rmSync`). Frontend uses Testing Library + `vi.stubGlobal('fetch', mockFetch)`. Playwright smoke specs use `page.waitForResponse()` / `page.route()`. The **host agent (bash) and physical rotation are not unit-testable** — manual verification only. Commands: `npm run test --workspace backend`, `npm run test --workspace frontend`, `npx playwright test`.

## Design Decisions

- **Button behavior**: one press cycles **90° clockwise** through `0 → 90 → 180 → 270 → 0`. The backend endpoint is a simple **absolute setter** (`POST /api/display/rotation` with `{ rotation: 0|90|180|270 }`); the frontend reads current state on mount via `GET` and computes the next value on click. This keeps the endpoint trivially testable and the cycling logic in one place. (Alternative considered: a 4-way orientation menu — rejected as heavier UI for a kiosk; revisit if needed.)
- **State ownership**: `rotation.json` in the pi-owned bind-mounted host directory is the single source of truth for "desired rotation," written by the backend and read by both the host agent (to apply) and the frontend `GET` (to seed the button). The host agent also rewrites kanshi config so the choice survives reboot.
- **Access model (DD-7)**: the agent must run as a systemd **user** service (it needs the labwc session's `WAYLAND_DISPLAY`/`XDG_RUNTIME_DIR`), but a non-root user cannot traverse the root-`0700` `/var/lib/docker/volumes/` tree where the named volume's `_data` lives. So `rotation.json` lives in a **dedicated bind-mounted host directory owned by the `pi` user** (`./chores-rotation` next to `docker-compose.yml`, mounted at `/rotation`), kept separate from the SQLite `chores-data` named volume. The container `node` user is uid 1000 — the same uid as the Pi `pi` user — so the backend can atomically write the file and the user service can read it with no `sudo`, no ACL hacks, and no second forwarding service. This also decouples rotation state from the DB volume, so the DD-6 name-pin step no longer risks the rotation file (and the chores DB volume model is left unchanged — no DB migration). (Rejected: Option 2 root system-service copy-hop — extra service + poll loop; Option 3 `setfacl` on the volume — fragile, Docker may reset ACLs.)
- **Button orientation set (DD-8)**: the app is portrait-only by explicit design — `frontend/src/index.css` ships `#rotate-overlay` (full-screen `z-9999`) shown via `@media (orientation: landscape) and (max-height: 500px)`, a tested guard (`e2e/smoke.spec.ts`). The shipped panel is landscape-native rotated to portrait (kanshi `transform 90`), so transforms `0`/`180` render **landscape** and on a small kiosk panel (rotated height < 500px) the overlay covers the whole screen — including the rotate button — locking the user out. Therefore the **button cycles only between the two portrait transforms `90 ↔ 270`** (a 180° portrait flip, useful for an inverted mount). The wire contract / type `DisplayRotation = 0|90|180|270` and `VALID_ROTATIONS` are kept intact (backend + agent still accept all four for manual override), but the button's click handler never produces `0`/`180`. (Rejected: Option 2 keep 4-way + verify panel ≥ 500px or weaken the tested overlay guard — panel-dimension-dependent and defeats the portrait-only design; Option 3 re-scope to a bespoke two-state toggle — needlessly tears down the wire contract.)
- **Transform vs matrix**: the wire contract carries only the integer transform (`0/90/180/270`). The host agent owns the transform→token→`calibrationMatrix` lookup (from `deploy/pi/README.md`) via a single `case` table (DD-1). **Integer `0` maps to the keyword `normal`** for both `wlr-randr --transform` and the kanshi `transform` token — both tools reject literal `0`. Frontend/backend never touch the token or the matrix; the integer stays on the wire only.
- **Security**: the endpoint inherits the app's current no-auth posture. The host agent **must allowlist-validate** the transform before interpolating it into `wlr-randr`/`sed` (injection guard). An optional shared-secret header is Step 6 (off by default).
